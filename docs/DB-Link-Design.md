# DB Link Architecture — Dynamic Price List ↔ ERP สินค้า

เอกสารออกแบบการเชื่อมต่อระหว่าง "ตารางราคา" กับ "ฐานข้อมูลสินค้า (ERP)" ผ่าน API local
ออกแบบทีละส่วน · อนุมัติก่อนเขียนโค้ด · ทำทีละ field

## กติกาที่ยืนยันแล้ว (ห้ามเปลี่ยนโดยไม่แจ้ง)
1. กุญแจลิงก์ = รหัสสินค้า 13 หลัก (เช่น 0010230314002). DB มาทาง API จาก server local.
2. การผูกลิงก์ = ผู้ใช้ทำเองด้วยมือ "ครั้งแรกครั้งเดียว" ต่อจุด (ไม่ auto-match).
3. สองทาง: ดึงมาดู (read) ทุกฟิลด์ที่ผูก · เขียนกลับ (write) ได้เฉพาะ "ราคาขาย 1–5" เท่านั้น ฟิลด์อื่นห้ามแก้.
4. คอลัมน์: ผู้ใช้ระบุเองว่าคอลัมน์ไหนดึงฟิลด์ไหน (ครั้งแรก). หน้าที่ระบบ = เสนอ "ชื่อฟิลด์ที่ถูกต้อง" ให้เลือก.
5. มีเคสพิเศษ: ช่องเดียวดึงราคาจากรหัสสินค้า/ฟิลด์ที่เจาะจง (ad-hoc cell link).

## ส่วนที่ 1 — Data Model & Schema (รออนุมัติ)

### 1.1 ระดับการลิงก์ (3 ชั้น, ใช้ลำดับความสำคัญ override → row → manual)
- **Cell link (สูงสุด):** `cellLinks[r][c] = { code, field }` — ช่องนี้ดึง field นี้ จากรหัสนี้ โดยเฉพาะ
- **Row link:** `rowLinks[r] = { code }` — ทั้งแถวผูกรหัสนี้ คอลัมน์ดึง field ตาม columnMap
- **Manual:** ไม่มีลิงก์ = ค่าที่พิมพ์เอง (เหมือนปัจจุบัน)

### 1.2 ตารางจับคู่คอลัมน์ (ตั้งครั้งแรก, ใช้ทั้งชีต)
`columnMap[colIndex] = { field, mode:'read'|'write' }`
- mode read = ดึงมาแสดง ล็อกไม่ให้พิมพ์ทับ
- mode write = อนุญาตเฉพาะ 5 ฟิลด์ราคาขาย (ดึงได้ + ส่งกลับได้)

### 1.3 แค็ตตาล็อกชื่อฟิลด์ DB (ระบบเสนอให้เลือก)
อ่านอย่างเดียว (read-only):
- identity: `code13`, `name`(รายละเอียดสินค้า), `status`, `brandCode`, `brandName`, `group`, `productType`
- แตกจากชื่อ: `size`(ขนาด), `ply`(ชั้น/PR), `model`(รุ่น), `spec`(106/104Q), `dot`, `lot`, `week`
- ต้นทุน: `costStandard`, `costAverage`(เฉลี่ยจากซื้อ), `costLatest`
- สต็อก: `qtyOnHand`(รวม), `qtyByWarehouse[wh]`, `lotBalance`
- หน่วย/ระดับ: `unit`, `reorderPoint`, `minLevel`, `maxLevel`

อ่าน+เขียน (เฉพาะ 5 — เป็น write ได้):
- `salePrice1`, `salePrice2`, `salePrice3`, `salePrice4`, `salePrice5`

### 1.4 สถานะ/เมตา ต่อจุดลิงก์
`linkMeta[r] = { lastPulled, lastPushed, dbStatus:'ok|missing|inactive', err }`
- ใช้แสดง marker เขียว/แดง (ทำไว้แล้ว) + เวลาที่ดึงล่าสุด

### 1.5 หลักการกันพลาด
- read field = ล็อกช่อง (อ่านอย่างเดียว) กันพิมพ์ทับแล้วเพี้ยน
- write จำกัด whitelist 5 ฟิลด์ระดับโค้ด — แม้ตั้ง map ผิดก็ดันค่าฟิลด์อื่นไม่ได้
- ทุกการ push เก็บ audit (code, field, old→new, เวลา, ผู้ทำ)

## ส่วนที่ 3 — UI ตั้งค่า/ผูกข้อมูล (admin · ครั้งแรก · รออนุมัติ)

### 3A — ผูกคอลัมน์กับฟิลด์ DB (column binding)
- admin คลิกขวาหัวคอลัมน์ → "ผูกกับฟิลด์ DB…"
- เปิดรายการแค็ตตาล็อกฟิลด์ จัดกลุ่ม (identity / ขนาด / ทุน / สต็อก / ราคา)
- เลือกฟิลด์ → ระบบ set mode อัตโนมัติ: 5 ราคา = write, อื่น = read + ล็อกช่อง
- หัวคอลัมน์ที่ผูกแล้ว: badge 🔗ชื่อฟิลด์ · read=ไอคอนล็อก · write=ไอคอนดินสอ
- เก็บใน columnMap

### 3B — ผูกแถวกับรหัสสินค้า (row link) — ยืนยัน
- ค้นหารหัส/ชื่อสินค้าในตารางค้นหา → double-click เพื่อผูกรหัสนั้นเข้าแถว
- กฎ: รหัสที่ผูก "ทั้งแถว" แล้ว ห้ามผูกซ้ำกับแถวอื่น (unique ต่อแถว)
- เก็บใน rowLinks · (ตอนนี้ใช้รายการ mock ก่อน → ดึงจริงตอนต่อ API)

### 3C — ผูกช่องเฉพาะกิจ (ad-hoc cell link) — ยืนยัน
- เป็นข้อยกเว้นของกฎ unique: cell-link สามารถใช้รหัส/ฟิลด์ข้ามสินค้าได้ในบางกรณี
- ใช้เมื่อช่องเดียวต้องดึงจากรหัส/ฟิลด์ที่ต่างจากทั้งแถว

### 3A — ผูกคอลัมน์ — ยืนยัน + กฎ
- ผู้ใช้ผูกเองทุกครั้ง + มี "เทมเพลตเริ่มต้น" ให้แก้ (ทุน→ทุน, B/A/S→ราคา1/2/3 ฯลฯ)
- กฎผูกทั้งแถว: ทุกคอลัมน์ต้องเป็นฟิลด์ของสินค้าตัวที่ผูกเท่านั้น ห้ามปนสินค้าอื่น
  (การปนข้ามสินค้าทำได้เฉพาะผ่าน cellLinks เท่านั้น)

### 3D — ตัวจัดการชั้นกลาง — ยืนยัน

## ส่วนที่ 4 — API Contract & Sync (รออนุมัติ)

### 4.1 Endpoints (ระหว่าง Staging Layer ↔ DB local)
1. ค้นหา/รายการสินค้า (ใช้ใน 3B picker + 3D manager)
   GET /products?q=&group=&brand=&status=&page=&limit=
   → [{ code13, name, status, brandCode, brandName, group }]
2. รายละเอียดสินค้าเต็ม (ตอนผูก + รีเฟรช)
   GET /products/{code13}
   → { identity..., cost*, qtyOnHand, qtyReserved, qtyAvailable,
        dotWeeks:[{dot,week,qty}], imageUrl, salePrice1..5 }
3. แก้ราคาขาย (เขียนกลับ — เฉพาะ 5 ช่อง)
   PUT /products/{code13}/prices  body:{ salePrice1..5 (ส่งเฉพาะที่เปลี่ยน) }
   → { ok, updated:{...}, serverTime }
4. รีเฟรชหลายตัว (อัปเดตทั้งชีต)
   POST /products/batch  body:{ codes:[...] } → [ราย product]

### 4.2 Config การเชื่อม — ยืนยัน + ขยาย (รองรับ remote)
- ใช้งานหลักบน notebook · รองคือ tablet/มือถือ · ต้องรองรับเชื่อมเน็ตภายนอกด้วย
- base URL ตั้งค่าได้: local IP (ในวงแลน) หรือ domain ภายนอก (HTTPS)
- Auth: ในวงแลนอาจไม่ต้อง · เชื่อมภายนอก = ต้องมี token (เตรียมช่อง token + เปิด/ปิดใน settings)
- เผื่อ CORS/HTTPS + timeout/retry สำหรับเน็ตภายนอกที่ช้า/หลุด
- ไม่มี endpoint เขียนอื่นนอก /prices (กฎเหล็ก)

### 4.3b รูปสินค้า — ✅ ใช้ URL (สรุป 17 มิ.ย.)
- DB ส่ง URL รูป → ImageResolver แค่โหลดจาก URL (ไม่ใช่ chunks/base64)

### 4.4 Sync strategy — ยืนยัน
- Adapter pattern (Mock → Http) · Cache-first: โหลด cache ก่อน แล้วรีเฟรชเบื้องหลัง
- ดึงเมื่อ: เปิดชีต (batch แถวที่ผูก, เบื้องหลัง) + ปุ่มรีเฟรช + หลัง push ราคา
- read fields ไม่ทับค่า manual ของช่องที่ไม่ผูก

## ส่วนที่ 5 — เขียนราคากลับ DB (5 ช่อง) + Audit (รออนุมัติ)

### 5.1 ช่องที่เขียนได้
- เฉพาะคอลัมน์ที่ columnMap mode='write' และ field ∈ {salePrice1..5}
- whitelist ระดับโค้ด: แม้ map ผิดก็ push ฟิลด์อื่นไม่ได้

### 5.2 Flow การ push
1. user แก้ราคาในช่อง write → ช่องขึ้นสถานะ "แก้แล้ว ยังไม่ส่ง" (dirty, สีต่าง)
2. กด "ส่งราคาเข้า DB" (รายแถว / เลือกหลายแถว / ทั้งชีต)
3. กล่องยืนยัน: แสดงตาราง รหัส · ฟิลด์ · ค่าเดิม→ค่าใหม่ ให้ตรวจก่อน
4. PUT /prices ทีละรหัส (หรือ batch) → แสดงผลรายช่อง: สำเร็จ ✓ / ล้มเหลว ✗ + เหตุผล
5. สำเร็จ → ล้าง dirty, อัปเดต cache, บันทึก audit

### 5.3 Audit log (ฝั่งเรา) + หน้าต่างแยก
- ทุกการส่ง: { เวลา, code13, name, field, ค่าเก่า, ค่าใหม่, ผู้ทำ, อุปกรณ์, ผล }
- หน้าต่าง Audit แยก (ฟังก์ชัน/ไอคอนของตัวเอง):
  - ค้นหา/กรองด้วย: รหัสสินค้า · ชื่อสินค้า · คำค้น · วันที่แก้ · วันที่อัปเดต · ช่วงวันเวลา
  - ดูประวัติรายรหัสว่าปรับแก้ไปแบบไหนบ้าง (timeline ค่าเก่า→ใหม่)
- หน้าต่าง Update แยก: อ้างอิงการกด "อัปเดต" — ดูค่า ก่อน/หลัง รายครั้ง
  + "ตั้งเวลาอัปเดต" (schedule push ตามเวลาที่กำหนด)

### 5.2 Flow การ push — ยืนยัน + 2 โหมด
- ส่งได้เฉพาะ ADMIN เท่านั้น (perm)
- 2 โหมดอัปเดต: (1) อัปเดตทันที (2) อัปเดตตามเวลาที่กำหนด
- Schedule: server เป็นตัวส่งเอง (มี server แยกรันอยู่แล้ว) — ไม่ต้องเปิดโปรแกรมค้าง

### 5.6 สัญลักษณ์การปรับราคา (price-change indicator)
- เมื่อสินค้า "กำลังจะอัปเดต" หรือ "อัปเดตแล้ว" → แสดงสัญลักษณ์ที่ช่องราคา + ลูกศรขึ้น/ลง
- hover → บอกว่าราคาปรับไปเท่าไหร่ (delta)
- ช่องที่เป็นโค้ดราคา (ค้าส่ง) → แสดง delta เป็นโค้ดด้วย มี +/- นำหน้า ทั้งตัวเลขและตัวอักษร
- อายุสัญลักษณ์ 7 วัน นับจากวันเวลาที่อัปเดต → หลังจากนั้นหายไป
- เก็บ: priceChange[code13] = { field, old, new, dir, ts } (ฝั่งเรา)

## ส่วนที่ 8 — การแสดงราคา + ประวัติ + กฎคำนวณ (ใหม่ · รออนุมัติ)

### 8.1 คลิกที่ราคา / โค้ดราคาค้าส่ง → ป๊อปอัปประวัติ
- แสดงราคาย้อนหลัง 1 รายการ + ระบุ "วันที่สุดท้ายที่ใช้"
- แสดง: ราคาขาย + ราคาค้าส่ง(โค้ด) แบบราคาเครดิต
- แสดง: ราคา เครดิต+VAT (คำนวณให้ ตามกฎที่ตั้งไว้)

### 8.2 กฎคำนวณราคา (pricing rules)
- ราคาเครดิต+VAT = บวกเพิ่มจากเครดิตตาม "กฎ" ที่ตั้งไว้
- กฎมีหน้าต่างแยก (จัดการเต็ม) + โหมดหลักอยู่ในหน้าตั้งค่า
- เก็บ: pricingRules = { vatRate, creditMarkup, rounding, ... }

### 8.3 ประวัติราคา (ฝั่งเรา) — ยืนยัน
- เก็บจาก audit เฉพาะราคาที่ "อัปเดตเข้า DB จริงแล้ว"
- เก็บย้อนหลัง 3 รอบ (แสดงในป๊อปอัป 1 รายการล่าสุด + วันที่ใช้ล่าสุด)

### 8.4 กฎเครดิต+VAT — ยืนยัน
- VAT = 7% (ตอนนี้) · แก้ค่า VAT ได้ในหน้าตั้งค่า (เผื่ออนาคต)
- ปัดเศษ "ขึ้นทีละ 10.00" (ceil ไปขั้นถัดไปของ 10): 11.22→20.00, 100.01→110.00, 100.00→100.00
- เก็บ: pricingRules = { vatRate:0.07, roundStep:10, roundMode:'ceil', creditMarkup:... }

### 8.5 โค้ดราคาค้าส่ง +/- — ยืนยัน
- ใช้ระบบเข้ารหัสราคาเป็นตัวอักษรเดิมของโปรแกรม
- เติม +/- นำหน้าโค้ดเพื่อบอกส่วนต่างเท่านั้น

## ส่วนที่ 6 — นิยามกฎ auto (ส่วนสุดท้าย · ออกแบบร่วมกัน)

### 6.1 Input (จาก clean model)
- qtyOnHand, qtyReserved, qtyAvailable(=onHand−reserved), setSize
- incoming (จำนวน PO/ของกำลังเข้า), needsWithdraw(flag)
- flags จาก enrich: special / clearance / rare / กำหนดเอง

### 6.2 ไอคอนสถานะ (ปรับใหม่ 17 มิ.ย. · ปรับแต่งได้ในตั้งค่า)

กลุ่ม STOCK = จุดสี 1 จุด (เลือกสถานะเดียว ตามลำดับ):
| จุด | สถานะ | เงื่อนไข |
|---|---|---|
| 🔴 แดง | หมด | qtyOnHand ≤ 0 |
| 🟠 ส้ม | ติดจองจนสุทธิ ≤ 0 | onHand > 0 และ qtyAvailable ≤ 0 |
| 🟣 ม่วง | ติดจอง: รวมจองครบชุด แต่สุทธิไม่ครบ | onHand ≥ setSize และ 0 < qtyAvailable < setSize |
| 🟢 เขียว | ครบชุด (สุทธิยังครบ) | qtyAvailable ≥ setSize |
| ⚪️ (รอสรุป) | ของไม่ถึงชุดจริง (gross < setSize, ไม่ใช่เพราะจอง) | 0 < onHand < setSize |

กลุ่ม ATTRIBUTE = ไอคอนสัญลักษณ์ (แสดงเพิ่มได้หลายอัน รวมเพดาน 4):
| ไอคอน | สถานะ | ที่มา |
|---|---|---|
| 🚚 รถบรรทุก | ของกำลังเข้า/รอดึงจากโกดัง | incoming > 0 |
| ➕ บวกเขียว | ต้องเบิก | needsWithdraw |
| ⭐ ดาวเหลือง | ราคา NET พิเศษ | flag special |
| 🫗 เทน้ำส้ม | ราคาโละ | flag clearance |
| 💎 เพชรฟ้า | หายาก | flag rare |
| ... | เพิ่มเองได้ | enrich/manual |

- ไอคอน/สี/ความหมายทั้งหมด แก้ได้ใน Tab "🏷️ ไอคอนสถานะ" (หน้าตั้งค่า)
- 1 จุดสีสต็อก + ไอคอน attribute หลายอัน รวมไม่เกิน 4 ต่อช่อง

### 6.3 ลำดับความสำคัญ (priority) — ปรับเองได้
- ตั้ง priority ทุกสถานะเองได้ (ลำดับ 1 → ท้ายสุด)
- ลากย้ายขึ้น/ลงในหน้าตั้งค่า (Tab ไอคอนสถานะ)
- เมื่อเข้าหลายเงื่อนไข แสดงตามลำดับ priority สูงสุด 4 อัน
- mixed: ไอคอน manual ที่ pin แทรกตามตำแหน่งที่กำหนด

### 6.4 ทุกสถานะนิยาม/ตั้งค่าได้เอง (ยืนยัน)
- ทุกสถานะ (จุดสีสต็อก + ไอคอน attribute + ที่สร้างใหม่) แก้ได้ครบ:
  เงื่อนไข · ไอคอน · สี · ป้ายชื่อ · ข้อความ popup · priority · เปิด/ปิด
- เก็บ: statusDefs = [ { key, label, icon, color, cond, priority, enabled, popup } ... ]

### 6.5 setSize — ลำดับการตัดสิน (hierarchy)
1. กำหนดย่อยรายสินค้า (enrich[code13].setSize) — สูงสุด (รวม "ไม่ต้องกำหนด" ได้)
2. กฎรายหมวดสินค้า (categoryRules[group]) — อ้างฟิลด์ "หมวดสินค้า" เป็นหลัก
   - ต่อหมวด: ต้องเช็คชุดไหม? + ค่าชุด (2/4/กำหนดเอง)
3. ค่า default = 4 (ถ้าไม่มีระดับบนกำหนด)
- สินค้าในหมวดที่ตั้งค่าไว้ ยัง override ย่อยรายตัวได้ (2/4/ไม่ต้องเช็ค)
- เก็บ: categoryRules[group]={ checkSet:bool, setSize:n } · enrich[code].setSize=n|null|'none'

## ส่วนที่ 7 — Auth / Users / Permissions / Device Log (ใหม่ · รออนุมัติ)

### 7.1 แยกบทบาท (RBAC ขยายได้)
- อย่างน้อย: admin / user · ออกแบบ role เพิ่มได้ในอนาคต
- ผูกสิทธิ์รายฟังก์ชัน (permission granular):
  ดูชีต · แก้ราคา · ส่งราคาเข้า DB · ผูกลิงก์(3A-C) · จัดการชั้นกลาง(3D)
  · จัดการไอคอน · จัดการผู้ใช้ · ดู audit · ตั้งเวลาอัปเดต ฯลฯ

### 7.2 Login & ตัวตน — ✅ ใช้ของเดิม (ยืนยัน 17 มิ.ย.)
- โปรแกรมมีชื่อผู้ใช้/login อยู่แล้ว → ใช้เป็น "ตัวยืนยันตัวตน" (authentication) เลย
- ไม่ต้องทำ login/hash ใหม่ — ลดขั้นตอนมาก
- เราทำแค่ "Permission Layer" มาครอบ (ขนานกับ Enrichment Layer ของสินค้า):
  ดึงรายชื่อผู้ใช้ → กรอง + ใส่ role/permission ฝั่งเรา → ส่งให้โปรแกรมใช้
- เก็บ: permStore[username] = { role, perms:{...} } (ฝั่งเรา ไม่แตะระบบ login เดิม)

### 7.3 Device log (คงเดิม)
- ทุก session: เวลา, username, role, อุปกรณ์/OS, จอ, ภาษา, timezone, IP(ถ้าได้), ผล

### 7.5 ใครส่งราคา/ทำ action ไหนได้ — คุมด้วย permStore
- ตรวจ permission ก่อนทุก action (ส่งราคา/ผูกลิงก์/จัดการชั้นกลาง/จัดการผู้ใช้ ฯลฯ)

### 7.4 ผูกกับฟังก์ชันอื่น
- ส่งราคา/ผูกลิงก์/จัดการชั้นกลาง = ตรวจ permission ก่อนทุกครั้ง
- "ผู้ทำ" ใน audit = username ปัจจุบัน + device ที่ใช้

## ส่วนที่ 2.5 — DB Staging & Enrichment Layer (ชั้นกลาง · สำคัญ · รออนุมัติ)

ปัญหา: DB ยังไม่มีบางฟิลด์ (เช่น setSize 2/4) และ shape ของ DB จะเปลี่ยนได้
แนวคิด: แทรก "ชั้นกลาง" ระหว่าง API ↔ โปรแกรมหลัก เพื่อ กรอง + เติมแท็ก + แปลงรูป
ก่อนส่งข้อมูลสะอาดให้โปรแกรมหลักใช้ (decouple DB จาก logic — รองรับอนาคต)

```
[DB server local] --API--> [Staging Layer] --clean model--> [โปรแกรมหลัก/ตารางราคา]
                            ├─ 1. Pull: ดึง raw ตาม code13
                            ├─ 2. Filter: กรองสินค้า (กลุ่ม/ยี่ห้อ/สถานะ active)
                            ├─ 3. Enrich: เติมแท็กที่ DB ไม่มี เช่น
                            │      setSize(2/4), flags(พิเศษ/โละ/หายาก/ต้องเบิก)
                            │      — เก็บ override ในระบบเรา ผูกกับ code13
                            ├─ 4. Normalize: แปลงชื่อฟิลด์ให้เป็น schema มาตรฐาน
                            └─ 5. Serve: ส่ง model สะอาด + เวอร์ชัน/เวลาดึง
```

- **Enrichment store** (ของเรา ไม่ใช่ DB): `enrich[code13] = { setSize, flags:{...}, note, ... }`
  แก้ผ่าน UI จัดการ (ส่วน 3) — ขยายเพิ่มฟิลด์ในอนาคตได้โดยไม่กระทบโปรแกรมหลัก
- **Clean model** ที่ส่งออก = ฟิลด์ DB (read-only) + ฟิลด์ enrich (ของเรา) รวมกัน
- โปรแกรมหลัก/กฎ auto อ่านจาก clean model อย่างเดียว (ไม่แตะ raw DB โดยตรง)

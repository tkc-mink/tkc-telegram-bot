# ระบบ Log User + ผูกอุปกรณ์ + OTP ครั้งแรก — แผนงาน (Device Registry & Access Control)

## สรุปความต้องการ (จากผู้ใช้)
1. เก็บ **log การใช้งานของ user ทุกคน** พร้อม **ชื่ออุปกรณ์** ที่ใช้แต่ละเครื่อง
2. **ส่งรหัส (OTP) ครั้งแรก** ก่อนเข้าแอปในทุกเครื่อง (ลงทะเบียนอุปกรณ์)
3. **แอดมิน** ดู log user + ลบอุปกรณ์ที่ user ใช้ได้
4. อุปกรณ์มี 2 แบบ:
   - **แบบ A — เครื่องของร้าน (shared):** login ได้หลาย user · ตั้ง **เวลา auto-logout** ได้
   - **แบบ B — เครื่องส่วนตัวพนักงาน (bound):** login ครั้งแรก → **ผูกกับ user คนนั้นถาวร เปลี่ยน user ไม่ได้**
5. **ลบอุปกรณ์แล้ว → เข้าไม่ได้อีก** ต้องลงทะเบียนใหม่ด้วย **OTP ครั้งแรก** (เหมือนเครื่องใหม่)

---

## สถาปัตยกรรม (Client + Server)

### ทำไมต้องมี server
- **OTP**: แอป (เว็บฝั่งหน้า) ส่ง SMS/LINE เองไม่ได้ — ต้องให้ server สร้าง+ส่ง OTP และตรวจ
- **ลบแล้วเข้าไม่ได้อีก (ข้ามเครื่อง)**: ต้องให้ server เป็น "ทะเบียนกลาง" ที่อนุมัติ/เพิกถอนอุปกรณ์ · ถ้าเก็บแค่ในเครื่อง ผู้ใช้ล้าง cache ก็ทะลุได้
- **Log รวมศูนย์ + แอดมินดูข้ามเครื่อง**: ต้องเก็บที่ server

### หน้าที่ฝั่งแอป (ผมทำได้)
- หน้าจอ **ลงทะเบียนอุปกรณ์ครั้งแรก** (กรอก user → ขอ OTP → กรอก OTP → ตั้งชื่ออุปกรณ์ → เลือกแบบ A/B)
- สร้าง **deviceId** (สุ่ม, เก็บในเครื่อง) + **deviceToken** (server ออกให้หลังยืนยัน OTP)
- **แบบ A:** หน้า login เลือก user ได้หลายคน + ตัวจับเวลา auto-logout
- **แบบ B:** ผูก user เดียว ล็อกไว้ เปลี่ยนไม่ได้
- ทุกครั้งที่เปิดแอป: ส่ง deviceToken ให้ server ตรวจ → ถ้าถูกเพิกถอน (ลบ) → เด้งกลับหน้าลงทะเบียน OTP ใหม่
- บันทึก **log เหตุการณ์** (login/logout/เปิดแอป) ส่งขึ้น server
- **หน้าแอดมิน:** ตารางอุปกรณ์+user+log · ปุ่มลบอุปกรณ์ (เรียก API เพิกถอน)

### สัญญา API ที่ต้องการจาก server (flag ใหม่)
| flag/endpoint | ใช้ทำอะไร | ส่งไป | ตอบกลับ |
|---|---|---|---|
| `OTP_REQUEST` | ขอ OTP ครั้งแรก | user | `{ok, ref}` (ส่ง OTP ไป SMS/LINE ของ user) |
| `OTP_VERIFY` | ยืนยัน OTP + ลงทะเบียนอุปกรณ์ | user, otp, ref, deviceName, deviceType(A/B) | `{ok, deviceToken}` |
| `DEVICE_CHECK` | ตรวจว่าอุปกรณ์ยังใช้ได้ | deviceToken | `{ok}` / `{revoked:true}` |
| `DEVICE_LIST` | (แอดมิน) รายการอุปกรณ์+user+log | adminToken | `{devices:[{id,user,name,type,lastSeen,logs}]}` |
| `DEVICE_REVOKE` | (แอดมิน) ลบ/เพิกถอนอุปกรณ์ | adminToken, deviceId | `{ok}` |
| `LOG_PUSH` | ส่ง log เหตุการณ์ | deviceToken, event, ts | `{ok}` |
| `XLOGIN` | (มีแล้ว) ดึงรายชื่อ user+password | — | `{data:[{user,...}]}` |

## โหมดทำงานได้ทันที (ถ้า server ยังไม่พร้อม) — Local-only (ชั่วคราว)
ระหว่างรอ server เพิ่ม endpoint ผมทำเวอร์ชัน **เก็บในเครื่อง** ก่อนได้ เพื่อทดสอบ UX:
- ทะเบียนอุปกรณ์ + log + แอดมินลบ = เก็บใน localStorage
- OTP = ใช้รหัสจำลอง (เช่น แอดมินตั้งรหัสกลาง) แทนการส่ง SMS จริง
- **ข้อจำกัด:** "ลบแล้วเข้าไม่ได้อีก" จะกันได้แค่ในเครื่องเดิม (ล้าง cache ทะลุได้) — ต้องมี server ถึงจะกันจริง

## คำถามที่ต้องเคลียร์ก่อนสร้าง
1. **OTP ส่งทางไหน?** SMS / LINE Notify / อีเมล / แอดมินแจ้งด้วยปาก? (กำหนดช่องทาง = กำหนดงาน server)
2. **server เพิ่ม endpoint ตามตารางข้างบนได้ไหม?** หรือให้ผมทำ **Local-only** ไปก่อน?
3. **"แอดมิน" คือใคร** — login แยก/PIN พิเศษ/user ที่ flag เป็น admin จาก XLOGIN?
4. รายชื่อ user มาจาก **XLOGIN** (ที่เคยให้มา) ใช่ไหม?
5. แบบ A (เครื่องร้าน) auto-logout — ตั้งกี่นาที (ค่าเริ่มต้น)?

---

## ลำดับงานแนะนำ
1. ตอบคำถาม 1–5 → ผมล็อกสเปก
2. สร้าง **หน้าลงทะเบียน OTP + เลือกแบบอุปกรณ์ + ตั้งชื่อ** (ฝั่งแอป)
3. สร้าง **แบบ A (หลาย user + auto-logout)** และ **แบบ B (ผูก user เดียว)**
4. สร้าง **หน้าแอดมิน: ตาราง log + ลบอุปกรณ์**
5. ต่อ API จริง (หรือ Local-only ก่อน) + ทดสอบ + smoke test
